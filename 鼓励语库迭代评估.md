# 鼓励语库全面评估与迭代方案

> 评估对象：`src/constants/encouragements.ts`、`src-tauri/migrations/002_encouragement_settings.sql`、`003_encouragement_level.sql`、`src/stores/encouragementStore.ts`、`src/stores/taskStore.ts`、`src-tauri/src/commands/encouragement.rs`、`src/views/EncouragementView.vue`、`src/views/DashboardView.vue`、`src/components/TaskItem.vue`、`src/App.vue`
> 评估日期：2026-07-19
> 技术栈：Vue 3 + Naive UI + Tauri + SQLite (sqlx)

---

## 一、现状梳理

鼓励语库当前由两套相互独立的文案源驱动。前端硬编码 10 条字符串数组（`constants/encouragements.ts`）服务于 toast 与仪表盘 banner；数据库 `encouragements` 表承载 19 条预设文案与用户自定义文案，服务于完成首任务的弹窗。两者文本无任何交集。

| 维度 | 前端硬编码源 | 数据库源 |
|------|------------|---------|
| 文案总量 | 10 条 | 19 条预设 + 用户自定义 |
| 数据结构 | `string[]` 纯文本 | `Encouragement`（id/text/category/level/created_at） |
| 分级 | 无 | 4 级（normal/advanced/highlight/celebration） |
| 触发位置 | TaskItem toast、DashboardView banner | App.vue NModal 弹窗 |
| 持久化 | 编译进 bundle | SQLite 表，支持增删 |

数据库源按连续天数分级，等级映射规则为：`streak >= 7 → highlight`、`>= 3 → advanced`、`否则 → normal`，全部根目标完成时触发 celebration。选择算法采用 SQLite `ORDER BY RANDOM()`，等级桶内均匀随机，缺失时按 `目标等级 → normal → 全库` 降级。

文案风格混合口语化（"三连击！"）、文学格言体（"日拱一卒，功不唐捐。"）、励志口号（"自律给我自由。"）三种，长度集中在 8-20 字，全部中文且无 emoji。

## 二、问题诊断

### 2.1 内容多样性维度

**D1. 文案总量严重不足**

数据库预设仅 19 条，按 4 级分布为 normal 4 条、advanced 6 条、highlight 5 条、celebration 4 条。用户在 normal 等级（最常见的连续 1-2 天场景）下仅有 4 条可选，按均匀随机抽取，3 次内重复概率达 17.5%（生日问题近似）。前端硬编码 10 条的重复概率更高，10 次抽取内重复几乎必然。

**D2. 两套文案源割裂**

`constants/encouragements.ts` 的 10 条与 DB 的 19 条预设**文本完全无交集**。用户在鼓励语库管理页添加的自定义文案，永远无法出现在 toast 和仪表盘 banner 中。这是 Sprint 4 前端硬编码 → Sprint 4 后端 DB → Sprint 5 分级演进的历史遗留，constants 未迁移到 DB。

**D3. 风格分布失衡**

数据库 19 条中文学格言体占约 40%（"星光不负赶路人""日拱一卒""种一棵树最好的时间是十年前"），这类文案高频出现会产生说教感与审美疲劳。口语化文案占比不足 30%，且多集中在 advanced 等级，normal 等级偏严肃。

**D4. 缺少挫折/安抚类文案**

当前 19 条全部为正向激励，无任何针对逾期、连续中断、任务跳过的安抚文案。用户在连续中断后重新开始时，仍只能抽到"连续三天打卡，习惯正在养成！"这类与当前状态错位的文案，产生违和感。

### 2.2 情境适配性维度

**S1. 分级维度单一**

仅按"连续完成天数"分级，未覆盖任务类型、时段、目标进度、用户情绪等情境维度。早晨 9 点完成与深夜 23 点完成触发同一套文案；学习类任务与健身类任务触发同一套文案；目标进度 10% 与 90% 触发同一套文案。

**S2. 触发场景覆盖不全**

| 场景 | 是否触发 | 备注 |
|------|---------|------|
| 完成今日首任务 | 是（modal） | 核心场景 |
| 每次完成任务 | 是（toast） | 与首任务 modal 双弹 |
| 进入仪表盘 | 是（banner） | 随机一条 |
| 全部目标完成 | 是（celebration modal） | 仅触发一次 |
| 单个目标完成 | 否 | 仅根目标全部完成才触发 |
| 逾期任务被补救 | 否 | 无正向强化 |
| 连续中断后回归 | 否 | 无特殊文案 |
| 任务跳过 | 否 | 无解释/安抚 |
| 每日首次打开应用 | 否 | 无晨间激励 |
| 周末/节假日 | 否 | 无情境文案 |

**S3. 连续天数阈值僵硬**

`streak >= 7 → highlight`、`>= 3 → advanced`、`否则 → normal` 三档阈值固定，无法适配"用户已坚持 30 天"这类长周期场景。streak=30 与 streak=7 触发同一 highlight 文案，缺乏对长期坚持的递进激励。

**S4. celebration 降级链不一致**

`random_celebration_encouragement` 在 celebration 等级缺失时降级到 highlight，而 `random_encouragement_by_streak` 降级到 normal。前者优先保证文案"分量"，后者优先保证可用性，设计意图未统一。

### 2.3 语言表达维度

**L1. 文案与产品调性脱节**

产品定位为个人目标管理工具，但文案普遍偏"鸡汤化"和"学生腔"（"学习是给自己最好的礼物""种一棵树最好的时间是十年前"）。对于使用该产品管理职业目标、健身计划、副业项目的成年用户，这类文案容易引发抵触。

**L2. 长度两极分化**

normal 等级文案偏短（8-12 字），celebration 等级文案偏长（14-20 字），但 modal 弹窗的视觉空间一致，导致 normal 文案在弹窗中显得空旷，celebration 文案在 toast 中被截断。

**L3. 缺少人称代入**

19 条文案全部使用第二人称（"你"）或无主语陈述句，无第一人称视角（"我今天又做到了"）。用户日记式的自我肯定文案缺失，无法满足"记录给自己的话"这类需求。

**L4. 无 emoji 与视觉符号**

文案全部纯文本，无 emoji 装饰。虽然符合"无 emoji"的项目约定，但在 celebration 场景下缺少视觉庆祝元素（如🎉、✨），与"庆祝"的情感强度不匹配。

### 2.4 用户体验维度

**U1. 完成首任务时双弹窗冲突**

`TaskItem.doComplete` 先调 `taskStore.completeTask`（内部触发 modal），再调 `message.success(randomEncouragement())`（toast）。首任务完成时 modal 与 toast 同时出现，且文案来自不同源，用户注意力被分散，体验割裂。

**U2. 无去重与冷却机制**

每次完成任务都触发 `randomEncouragement()`，无"今日已展示"标记，无最近 N 条历史记录。用户连续完成 5 个任务时，toast 可能重复弹同一条文案。

**U3. celebration 缺少仪式感**

全部目标完成是产品中最具情感价值的时刻，但当前仅展示一个 NModal 弹窗（图标 + 文案 + 按钮），无 confetti 撒花、无进度回顾、无成就解锁动画。与"庆祝"的情感强度严重不匹配。

**U4. 鼓励语库管理页信息过载**

`EncouragementView` 顶部统计卡 + 添加表单 + 4 级列表混合在一屏，用户添加自定义文案时需先理解 4 级含义。`LEVEL_META` 的 desc 字段（"普通/进阶/高亮/庆祝"）对用户而言抽象，缺少"什么时候会触发"的具体说明。

**U5. 无用户偏好控制**

用户无法关闭鼓励语、调整弹窗频率、选择偏好风格（如"多给我文学类""少给我口号类"）。对不喜欢鼓励语的用户，每次完成任务都弹 toast 是干扰。

### 2.5 技术架构维度

**T1. 前端硬编码未迁移**

`constants/encouragements.ts` 应在 DB 化后废弃，但当前仍被 `TaskItem` 和 `DashboardView` 引用。维护时需同时改两处，且 DB 的新增文案不会反映到 toast/banner。

**T2. 选择算法无权重**

`ORDER BY RANDOM()` 是均匀随机，无法实现"新文案优先""低频文案优先""用户收藏优先"等加权策略。用户添加的自定义文案与预设文案同等概率，无法突出个人文案。

**T3. 无展示历史记录**

数据库无 `encouragement_show_log` 表，无法查询"最近 7 天展示了哪些文案""哪条文案展示次数最多"。无法基于历史做去重与偏好学习。

**T4. streak 算法与文案解耦**

`get_streak` 返回 `current_streak/longest_streak/completed_today`，但 `random_encouragement_by_streak` 仅用 `current_streak` 做分级。`longest_streak`（历史最长）这一有价值的信号未被利用，如"你曾坚持 21 天，这次能超越吗"这类文案无法触发。

### 2.6 管理功能维度

**M1. 完全缺失编辑功能**

后端 `encouragement.rs` 仅提供 `add_encouragement` 与 `delete_encouragement` 两个写命令，**没有 `update_encouragement` 命令**。Store 层 `encouragementStore.ts` 同样只有 `add` 与 `remove`，无 `update` 方法。前端 `EncouragementView.vue` 列表项仅渲染删除按钮（第 252-264 行），无编辑入口。用户添加自定义文案后若想修改一个字、调整一个等级，只能删除重建，丢失 `created_at` 与 `id`，且无法保留展示历史关联。

**M2. 预设文案完全只读**

后端 `delete_encouragement` 对 `category == "preset"` 直接返回 `AppError::Business("预设鼓励语不允许删除")`（第 70-72 行）。前端对预设文案不渲染删除按钮（第 253 行 `v-if="item.category === 'custom'"`）。用户若不喜欢某条预设文案（如觉得"鸡汤味"太重），无法删除、无法隐藏、无法禁用、无法改等级，只能被迫接受它在随机池中持续出现。

**M3. 无批量操作能力**

列表项无 checkbox，无全选/反选，无批量删除、批量改等级、批量导出选中。用户若积累了 50 条自定义文案想整理，只能逐条点击删除并逐条确认 Popconfirm，操作成本随数量线性增长。

**M4. 无搜索与筛选**

`EncouragementView` 顶部仅有添加表单，无搜索框、无按等级筛选、无按类别（预设/自定义）筛选。文案超过 30 条后，定位特定文案只能肉眼扫描四个等级分组列表。

**M5. 无排序控制**

列表按 `created_at` 升序固定排列（后端 `ORDER BY created_at`），无按字母、按等级、按创建时间倒序、按自定义顺序排列的选项。用户无法把最常用的文案置顶。

**M6. 无复制与去重检测**

无"复制文案"按钮快速基于已有文案创建新文案。后端无文案内容唯一性校验，用户可重复添加完全相同的文案，造成随机池中某文案权重虚高。

**M7. 无导入导出（鼓励语库级）**

仅 `backup.rs` 提供全量数据导入导出（仅 custom，preset 跳过），无鼓励语库独立的导入导出入口。用户无法便捷地与他人分享文案集，也无法从文本文件批量导入一批文案。

**M8. 删除无二次确认反馈**

`handleDelete` 虽用 `NPopconfirm` 二次确认，但删除后仅 `message.success("已删除")` 简短 toast，无文案内容回显。用户删除多条后无法确认删了什么，需翻列表核对。

**M9. 添加表单缺乏校验与预览**

`handleAdd` 仅校验非空，无长度下限（如至少 2 字）、无重复检测、无敏感词过滤。添加前无预览，用户不知道文案在列表中实际呈现的效果（字号、换行、与图标的搭配）。

**M10. 无分页或虚拟滚动**

列表全量渲染所有文案，无分页、无虚拟滚动。当文案库扩充至 80+ 条（P1-1 目标）后，单页渲染 80 个 DOM 节点 + 图标 + 标签，低端设备可能出现滚动卡顿。

## 三、迭代优化方案

按实施优先级分四档。P0 为架构一致性修复，P1 为体验核心提升，P2 为情境扩展，P3 为个性化机制。

### P0：架构一致性修复（预计 1 天）

#### P0-1 统一文案源，废弃前端硬编码

**问题**：`constants/encouragements.ts` 10 条与 DB 19 条预设割裂，toast/banner 永远只用硬编码。

**措施**：
1. 将 `constants/encouragements.ts` 的 10 条文案按等级导入 DB（建议全部归为 normal），通过一次性迁移脚本 `004_merge_frontend_encouragements.sql` 写入
2. 废弃 `DEFAULT_ENCOURAGEMENTS` 与 `randomEncouragement()`，`TaskItem.vue` 和 `DashboardView.vue` 改为调用 `encouragementStore.random()`
3. `encouragementStore.random()` 后端不可用时，降级为组件内缓存的 fallback 数组（保留 5 条应急文案）

**预期效果**：文案源统一，用户添加的自定义文案可出现在所有展示位，维护成本减半。

**涉及位置**：`src/constants/encouragements.ts`（废弃）、`src/components/TaskItem.vue` 第 90 行、`src/views/DashboardView.vue` 第 101 行、`src/stores/encouragementStore.ts`。

#### P0-2 修复首任务双弹窗冲突

**问题**：`TaskItem.doComplete` 在首任务完成时同时触发 modal 与 toast，文案来自不同源。

**措施**：`TaskItem.doComplete` 增加 `isFirstToday` 判断，首任务完成时跳过 toast，仅由 `taskStore.completeTask` 内部触发的 modal 承载鼓励语：

```ts
async function doComplete(qty: number) {
  const beforeDoneCount = taskStore.todayTasks.filter(t => t.status === "done").length;
  const updated = await taskStore.completeTask({ task_id: props.task.id, actual_qty: qty });
  goalStore.updateTaskLocally(updated);
  await goalStore.refreshProgressForGoalChain(updated.goal_id);
  if (beforeDoneCount > 0) {
    message.success(await encStore.random());  // 非首任务才弹 toast
  }
  emit("completed");
}
```

**预期效果**：消除首任务完成时的双弹窗冲突，用户注意力集中。

**涉及位置**：`src/components/TaskItem.vue` 第 88-92 行。

#### P0-3 修复 celebration 降级链不一致

**问题**：`random_celebration_encouragement` 降级到 highlight，`random_encouragement_by_streak` 降级到 normal，设计意图不统一。

**措施**：统一降级链为 `目标等级 → 上一等级 → normal → 全库`，celebration 降级路径改为 `celebration → highlight → advanced → normal → 全库`。在 `encouragement.rs` 中抽取 `random_with_fallback(levels: Vec<&str>)` 公共函数。

**预期效果**：降级行为可预测，避免 celebration 场景意外触发 normal 文案。

**涉及位置**：`src-tauri/src/commands/encouragement.rs`。

#### P0-4 引入展示历史与去重

**问题**：无去重，连续抽取可能命中同一条文案。

**措施**：
1. 新增 `encouragement_show_log` 表（`id, encouragement_id, shown_at, trigger_source`）
2. `random_encouragement_by_streak` 增加 `WHERE id NOT IN (最近 5 条展示记录)` 过滤
3. 候选不足时回退到全库随机

**预期效果**：连续 5 次内不重复，重复感知从 17.5%（3 次内）降至接近 0。

**涉及位置**：新增 migration `005_encouragement_show_log.sql`、`encouragement.rs` 查询逻辑。

#### P0-5 补齐编辑功能（最高优先级）

**问题**：后端无 `update_encouragement` 命令，store 无 `update` 方法，前端无编辑入口。用户改一个字都要删除重建。

**措施**：
1. 后端新增 `update_encouragement(id, text, level)` 命令，校验 `category != "preset"`（预设不可改文本），`level` 合法性校验，`text` 非空且长度 2-100
2. store 新增 `update(id, text, level)` 方法，调用后本地同步更新 `list`
3. 前端列表项增加编辑按钮（`mdi:pencil-outline`），点击弹出 `NModal` 编辑表单，含文本输入框（show-count maxlength=100）与等级下拉
4. 预设文案不渲染编辑按钮，仅自定义文案可编辑
5. 编辑成功后 toast 提示"已更新"

**预期效果**：补齐基础编辑能力，用户修改文案无需删除重建，保留 `id` 与 `created_at`。

**涉及位置**：`src-tauri/src/commands/encouragement.rs`、`src-tauri/src/lib.rs` 命令注册、`src/api/encouragement.ts`、`src/stores/encouragementStore.ts`、`src/views/EncouragementView.vue` 第 252-264 行列表项。

#### P0-6 删除反馈增强（直接物理删除）

**问题**：`handleDelete` 删除后仅 `message.success("已删除")` 简短 toast，无文案内容回显，用户删除多条后无法确认删了什么。

**措施**：
1. 保留现有 `NPopconfirm` 二次确认与物理删除（`DELETE FROM encouragements WHERE id = ?`），不引入软删除、撤销、回收站机制
2. 删除成功 toast 回显被删文案前 12 字：`message.success(\`已删除："${text.slice(0, 12)}${text.length > 12 ? '...' : ''}"\`)`
3. 列表项删除按钮在 `loading` 期间禁用，防止连点导致重复请求
4. 删除最后一项后该等级分组自动折叠或显示 `NEmpty`

**预期效果**：删除反馈清晰可追溯，操作防抖，交互成本不增加。

**涉及位置**：`src/views/EncouragementView.vue` 第 94-101 行 `handleDelete`、第 252-264 行列表项按钮。

### P1：体验核心提升（预计 2-3 天）

#### P1-1 文案库扩充至 80 条以上

**问题**：DB 预设仅 19 条，normal 等级仅 4 条，重复率高。

**措施**：按以下配比扩充至 80 条预设：
- normal：4 → 20 条（日常鼓励，口语化为主）
- advanced：6 → 20 条（3 天习惯养成，含成长视角）
- highlight：5 → 20 条（7 天里程碑，含长期坚持）
- celebration：4 → 10 条（全部完成，含仪式感）
- 新增 setback：10 条（连续中断后回归的安抚文案）

扩充方向参考：
- normal 新增：聚焦"开始行动"而非"坚持"，如"今天开始了，就赢过没开始的人"
- advanced 新增：聚焦"习惯形成"，如"第三天，大脑开始把它当成默认选项"
- highlight 新增：聚焦"身份认同"，如"坚持七天的人，已经在用行动定义自己"
- celebration 新增：聚焦"仪式感与回顾"，如"这段路你走完了，回头看都是自己的脚印"
- setback 新增：聚焦"重新开始的价值"，如"中断不等于归零，昨天积累的还在"

**预期效果**：normal 重复率从 17.5% 降至 2% 以内，风格分布更均衡。

**涉及位置**：新增 migration `006_expand_encouragements.sql`。

#### P1-2 引入挫折/安抚场景

**问题**：仅正向激励，无逾期、中断、跳过场景的文案。

**措施**：
1. 新增 `EncouragementLevel` 枚举值 `"setback"`（挫折安抚）
2. 触发时机：
   - 连续中断后首次完成任务（current_streak=1 且昨日有中断）
   - 逾期任务被完成时
   - 用户主动跳过任务后（可选，轻量 toast）
3. 文案风格：避免说教，聚焦"重新开始的价值"，如"昨天没完成没关系，今天开始就是新的连续"

**预期效果**：覆盖负向情绪场景，减少用户在中断后的流失。

**涉及位置**：`src/types/index.ts` 枚举、`encouragement.rs` 新增 `random_setback_encouragement` 命令、`taskStore.ts` 触发逻辑。

#### P1-3 celebration 仪式感增强

1. 引入轻量 confetti 动画（`canvas-confetti` 库，约 6KB gzip）
2. modal 内容增加"成就回顾"：展示本次征程的关键数据（总任务数、总耗时、连续天数峰值）

4. 庆祝文案后追加用户自定义"给自己的话"输入框（可选）

**预期效果**：提升全部完成场景的情感价值，增强用户分享意愿。

**涉及位置**：`src/App.vue` modal 区、新增 `src/components/CelebrationOverlay.vue`、`src-tauri/src/commands/` 新增成就查询命令。

#### P1-4 增加用户偏好设置

**预期效果**：满足不同用户偏好，减少对鼓励语反感的用户的干扰。

#### P1-5 批量操作能力

**预期效果**：50 条文案整理时间从逐条操作约 5 分钟降至 30 秒内。

**涉及位置**：`EncouragementView.vue` 列表项与顶部、`encouragement.rs` 新增批量命令、`encouragementStore.ts` 新增 `batchRemove`/`batchUpdateLevel`。

#### P1-6 搜索筛选与排序控制

4. 空结果时展示 `NEmpty` + "清除筛选"按钮

**预期效果**：80 条文案库中定位特定文案 < 2 秒，支持按场景快速整理。

**涉及位置**：`EncouragementView.vue` 新增筛选栏与 computed 过滤逻辑。

### P2：情境扩展（预计 3-4 天）

#### P2-1 多维度情境标签

**问题**：仅按连续天数分级，未覆盖时段、任务类型、进度等情境。

**措施**：扩展 `Encouragement` 结构，增加 `tags: string[]` 字段，支持多维度标注：

```ts
export interface Encouragement {
  id: string;
  text: string;
  category: "preset" | "custom";
  level: EncouragementLevel;
  tags: EncouragementTag[];  // 新增
  created_at: string;
}
export type EncouragementTag =
  | "morning"      // 晨间
  | "night"        // 深夜
  | "learning"     // 学习类任务
  | "fitness"      // 健身类任务
  | "work"         // 工作类任务
  | "milestone"    // 里程碑达成
  | "comeback"     // 回归鼓励
  | "progress_50"  // 进度过半
  | "progress_90"; // 即将完成
```

抽取时根据当前上下文匹配 tags，优先返回匹配度高的文案。

**预期效果**：文案与场景的匹配度从"按天数"提升到"多维匹配"，相关性显著增强。

**涉及位置**：`src/types/index.ts`、新增 migration `007_encouragement_tags.sql`、`encouragement.rs` 抽取逻辑、`EncouragementView` 添加表单增加 tags 多选。

#### P2-2 拓展触发场景

**问题**：仅首任务完成、每次完成、进 Dashboard 触发，场景覆盖不全。

**措施**：新增以下触发点：

| 场景 | 触发条件 | 展示形式 | 文案等级 |
|------|---------|---------|---------|
| 每日首次打开应用 | 当日首次 onMounted App.vue | banner | normal + morning tag |
| 单个目标完成 | `goal.percentage` 从 <1 变为 1 | toast | highlight |
| 逾期任务被完成 | `is_overdue` 的任务被 complete | toast | setback + comeback tag |
| 连续中断后回归 | current_streak=1 且昨日中断 | modal | setback |
| 周末完成 | 周六/日完成任务 | toast | normal + weekend tag |
| 进度过半 | goal.percentage 首次 >= 0.5 | toast | milestone + progress_50 tag |
| 即将完成 | goal.percentage 首次 >= 0.9 | toast | milestone + progress_90 tag |

**预期效果**：鼓励语覆盖用户全旅程关键节点，从"完成首任务才弹"扩展到"7 个关键场景"。

**涉及位置**：`taskStore.ts` 触发逻辑、`App.vue` onMounted、`goalStore.ts` 进度变化监听。

#### P2-3 时段感知文案

**问题**：早晨 9 点与深夜 23 点触发同一文案，情境错位。

**措施**：在抽取时传入当前时段（`morning` 6-11、`afternoon` 11-18、`evening` 18-23、`night` 23-6），优先匹配对应 tag 文案。深夜完成时触发"夜深了，注意休息"类关怀文案，而非"又是元气满满的一天"。

```ts
function getTimeTag(): EncouragementTag {
  const h = new Date().getHours();
  if (h >= 6 && h < 11) return "morning";
  if (h >= 23 || h < 6) return "night";
  return "morning"; // 默认
}
```

**预期效果**：文案与用户实际所处时段匹配，增强情感共鸣。

**涉及位置**：`encouragementStore.ts`、`encouragement.rs` 抽取函数增加时段参数。

#### P2-4 长周期递进激励

**问题**：streak >= 7 后全部归为 highlight，30 天与 7 天触发同一文案。

**措施**：扩展分级阈值，引入长周期等级：
- `streak >= 30 → master`（新增，大师级）
- `streak >= 14 → expert`（新增，专家级）
- `streak >= 7 → highlight`
- `streak >= 3 → advanced`
- `否则 → normal`

新增 master/expert 等级文案各 10 条，聚焦长期坚持的身份认同与里程碑回顾。

**预期效果**：覆盖长周期坚持场景，避免 7 天后激励强度停滞。

**涉及位置**：`src/types/index.ts` 枚举、migration 新增等级、`encouragement.rs` 分级逻辑。

#### P2-5 预设文案管理（隐藏/禁用/重置）

**问题**：预设文案完全只读，用户不喜欢的预设无法处理，只能被迫接受在随机池持续出现。

**措施**：
1. `encouragements` 表新增 `is_hidden BOOLEAN DEFAULT 0` 字段（仅作用于预设，用户可隐藏不喜欢的预设）
2. 所有查询增加 `WHERE is_hidden = 0` 过滤，隐藏的预设不参与随机抽取，也不在列表展示（除非切到"显示已隐藏"视图）
3. 列表项预设文案增加"隐藏"按钮（`mdi:eye-off-outline`），已隐藏文案在"回收站/已隐藏"分区可恢复
4. 新增 `reset_preset_encouragements` 命令：当迁移脚本更新预设文案后，用户可一键重置预设（先删除旧 preset，再插入新 preset，事务保证），保留用户的隐藏偏好
5. 预设文案仍不允许删除文本与改等级（保留系统预设完整性），但允许隐藏

**预期效果**：用户可控制预设文案的可见性，又不破坏系统预设的完整性，解决"被迫接受鸡汤"问题。

**涉及位置**：新增 migration、`encouragement.rs` 查询过滤与重置命令、`EncouragementView.vue` 预设文案操作按钮。

#### P2-6 复制文案与去重检测

**问题**：无复制按钮快速基于已有文案创建，后端无唯一性校验，可重复添加相同文案导致权重虚高。

**措施**：
1. 列表项增加"复制"按钮（`mdi:content-copy`），点击后调 `add(text, level)` 创建副本，toast 提示"已复制为新文案"
2. 后端 `add_encouragement` 增加去重校验：`SELECT COUNT(*) FROM encouragements WHERE text = ?`，若 > 0 返回 `AppError::Business("该鼓励语已存在")`
3. 前端添加表单失焦时调 `check_encouragement_duplicate` 接口预检，输入框下方实时提示"已存在相同文案"
4. 编辑时若改为已存在文案同样拦截

**预期效果**：消除重复文案导致的权重失衡，复制功能加速批量创建相似文案。

**涉及位置**：`encouragement.rs` 去重校验、`EncouragementView.vue` 复制按钮与输入预检。

### P3：个性化机制（预计 4-5 天）

#### P3-1 加权随机抽取

**问题**：`ORDER BY RANDOM()` 均匀随机，无法突出新文案、低频文案、用户收藏。

**措施**：引入加权抽取算法，权重计算公式：

```
weight = base_weight
       + new_bonus          // 新添加文案 7 天内权重 ×3
       + low_freq_bonus     // 展示次数 < 3 的文案权重 ×2
       + favorite_bonus     // 用户收藏的文案权重 ×2
       + style_match_bonus  // 匹配用户风格偏好的文案权重 ×1.5
       - recent_penalty     // 最近 5 次展示过的文案权重 ×0.1
```

SQLite 实现用 `ORDER BY (weight * RANDOM()) DESC LIMIT 1`，或在前端计算权重后抽取。

**预期效果**：新文案优先曝光，低频文案获得机会，用户收藏文案更常出现，重复感知进一步降低。

**涉及位置**：`encouragement.rs` 查询、新增 `encouragement_favorites` 表。

#### P3-2 用户收藏机制

**问题**：用户无法标记喜欢的文案，无法让喜欢的文案更常出现。

**措施**：
1. `EncouragementView` 列表每条增加收藏按钮（`mdi:heart-outline` / `mdi:heart`）
2. 新增 `encouragement_favorites` 表（`user_id, encouragement_id, favorited_at`）
3. 收藏文案在抽取时获得 `favorite_bonus` 权重
4. 鼓励语库管理页增加"我的收藏"筛选

**预期效果**：用户可主动塑造鼓励语库，收藏文案高频出现，情感连接增强。

**涉及位置**：新增 migration、`encouragement.rs`、`EncouragementView.vue` 列表项。

#### P3-3 展示反馈学习

**问题**：无展示历史，无法学习用户偏好。

**措施**：
1. 完善 `encouragement_show_log` 表，记录每次展示的 `encouragement_id/shown_at/trigger_source/user_dismissed`
2. `user_dismissed` 标记用户主动关闭 modal 的时间（快速关闭视为不感兴趣）
3. 抽取时降低 `user_dismissed` 率高的文案权重
4. 鼓励语库管理页增加"展示统计"：每条文案的展示次数、收藏次数、平均展示时长

**预期效果**：系统自动学习用户偏好，展示越久越贴合用户口味。

**涉及位置**：`encouragement_show_log` 表扩展、`App.vue` modal 关闭时记录 dismiss、`EncouragementView.vue` 统计页。

#### P3-4 longest_streak 信号利用

**问题**：`get_streak` 返回 `longest_streak` 但未被文案选择使用。

**措施**：当 `current_streak` 接近或超过 `longest_streak` 时，触发特殊文案：
- `current_streak === longest_streak` → "你正在追平自己的最佳记录"
- `current_streak > longest_streak` → "新纪录！你超越了曾经的自己"
- `current_streak < longest_streak * 0.5` → "你曾坚持 X 天，那段积累还在"

新增 longest_streak 触发的文案 5 条，等级归为 highlight。

**预期效果**：利用历史信号，在用户接近或超越自我时给予精准激励。

**涉及位置**：`encouragement.rs` 新增 `random_by_streak_with_best` 命令、`taskStore.ts` 传入 longest_streak。

#### P3-5 拖拽排序与自定义顺序

**问题**：固定按 `created_at` 排序，用户无法把常用文案置顶，无法按个人偏好排列。

**措施**：
1. `encouragements` 表新增 `sort_order INTEGER DEFAULT 0` 字段
2. 所有查询 `ORDER BY sort_order ASC, created_at ASC`
3. 前端列表项启用拖拽（参考 `GoalTreeNodeItem.vue` 拖拽实现，遵循项目约束：Tauri `dragDropEnabled: false`）
4. 拖拽 `@dragstart` 设置 `draggingId`，目标项 `@dragover.prevent` 计算上下分区，`@drop` 调 `reorder_encouragements(from_id, to_id, position)` 后端命令
5. 后端用事务批量更新 `sort_order`，保证排序连续无空洞
6. 拖拽时源项 `opacity-40`，目标项 `ring-2 ring-brand-400`（遵循项目视觉约定）

**预期效果**：用户可自由排列文案顺序，常用文案置顶，管理体验从"只读列表"进化为"可编排工作台"。

**涉及位置**：新增 migration、`encouragement.rs` 新增 `reorder_encouragements` 命令、`EncouragementView.vue` 列表项拖拽。

#### P3-6 独立导入导出与虚拟滚动

**问题**：仅 backup.rs 全量导入导出，无鼓励语库级独立入口；文案达 80+ 后全量渲染可能卡顿。

**措施**：
1. 新增 `export_encouragements(ids?: Vec<String>)` 命令：未传 ids 导出全部自定义，传 ids 导出选中，返回 JSON
2. 新增 `import_encouragements(items: Vec<AddEncouragementInput>, strategy: "skip"|"overwrite"|"rename")` 命令，事务保证，去重策略可选
3. 前端"导入"按钮接受 `.json` 文件，解析后预览待导入列表，用户确认后批量入库
4. 列表改用 `NVirtualList`（Naive UI 虚拟滚动）或分页（每页 20 条），仅渲染可视区域 DOM
5. 导入导出文件格式定义：`{ version: 1, items: [{ text, level }, ...] }`，便于跨用户分享文案集

**预期效果**：支持文案集分享与迁移，80+ 文案库滚动流畅，整理效率提升。

**涉及位置**：`encouragement.rs` 新增导入导出命令、`EncouragementView.vue` 改用虚拟列表与导入导出按钮。

### 实施建议

**P0 阶段**是架构一致性的必修课，尤其 P0-1（统一文案源）和 P0-2（修复双弹窗）直接影响用户感知，建议立即纳入下一次迭代。四项修改互不依赖，可并行开发。

**P1 阶段**是体验提升的核心。文案扩充（P1-1）建议与产品/运营协作打磨文案，避免纯工程视角产出"鸡汤感"文案。celebration 仪式感（P1-3）涉及新增依赖（canvas-confetti），需评估包体积影响。用户偏好设置（P1-4）是后续个性化机制的基础，应优先于 P2/P3 实施。

**P2 阶段**扩展情境维度，多维度标签（P2-1）是后续所有情境匹配的基础，应最先实施。拓展触发场景（P2-2）建议分批上线，避免一次性增加 7 个触发点导致鼓励语过载。

**P3 阶段**为个性化机制，依赖 P1-4 的用户偏好设置与 P0-4 的展示历史数据。加权随机（P3-1）和反馈学习（P3-3）需一定数据积累后才能体现效果，建议在 P2 上线后观察 2 周再推进。

### 验收标准

| 优化项 | 验收指标 |
|--------|---------|
| 文案源统一 | `constants/encouragements.ts` 废弃，toast/banner 走 DB |
| 双弹窗修复 | 首任务完成时仅 modal，无 toast |
| 降级链一致 | celebration 与 streak 降级路径遵循同一规则 |
| 展示去重 | 连续 5 次内不重复，重复率 < 5% |
| 文案扩充 | DB 预设 >= 80 条，normal >= 20 条 |
| 挫折场景 | 连续中断后回归时触发 setback 文案 |
| celebration 仪式感 | 全部完成时展示 confetti + 成就回顾 |
| 用户偏好 | 可关闭鼓励语、调频率、选风格 |
| 多维度标签 | 文案支持 tags 字段，抽取时按上下文匹配 |
| 触发场景覆盖 | 7 个关键场景均有鼓励语 |
| 加权随机 | 新文案 7 天内权重 ×3，收藏文案权重 ×2 |
| 收藏机制 | 用户可收藏文案，收藏文案高频出现 |
| 反馈学习 | 快速关闭的文案权重降低 |
| longest_streak 利用 | 接近或超越历史最佳时触发特殊文案 |
| 编辑功能 | 自定义文案可改文本与等级，保留 id 与 created_at |
| 删除反馈 | 删除 toast 回显文案前 12 字，按钮防抖，物理删除 |
| 批量操作 | 可多选、批量删除、批量改等级，事务保证 |
| 搜索筛选 | 支持文本搜索、类别与等级筛选，80 条库定位 < 2s |
| 预设文案管理 | 可隐藏不喜欢的预设，支持一键重置预设 |
| 复制与去重 | 可复制文案，重复添加被后端拦截 |
| 拖拽排序 | 可拖拽调整顺序，sort_order 字段生效 |
| 导入导出 | 独立 JSON 导入导出，支持分享文案集 |
| 虚拟滚动 | 80+ 文案库滚动流畅，仅渲染可视区域 |
