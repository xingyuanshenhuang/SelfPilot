# 日历视图全面评估与迭代方案

> 评估对象：`src/views/CalendarView.vue`（553 行单文件组件）
> 评估日期：2026-07-18
> 技术栈：Vue 3 `<script setup>` + Naive UI + UnoCSS + Iconify + date-fns + Tauri IPC

---

## 一、现状梳理

当前日历视图是一个高内聚单文件组件，承载月/周/日三种视图模式，核心能力包括任务展示、单任务与批量完成/跳过、依赖阻塞可视化、逾期标记。数据通过 `taskApi.listTasksByDateRange`（Tauri IPC）直接获取，复用 `goalStore` 刷新目标进度，不经过独立 store。

| 视图 | 布局 | 单元格高度 | 信息密度 | 交互入口 |
|------|------|-----------|---------|---------|
| 月 | 7列网格，周一起始 | 88px | 低（4个圆点+统计标签） | 点击切日视图 |
| 周 | 7列网格 | 280px | 中（任务名+状态图标） | 无 |
| 日 | 单卡片任务列表 | 自适应 | 高（完整字段+操作） | 全选/清空/批量完成/跳过 |

色彩体系以品牌蓝 `#3478f6` 为主色，任务状态用灰/橙/绿四色驱动图标与圆点。视觉反馈依赖 UnoCSS 原子类，scoped CSS 仅一处 `translateY(-1px)` 悬浮动效。

---

## 二、问题诊断

### 2.1 视觉设计维度

**V1. 月视图单元格空间利用率低**

`min-h-[88px]` 的单元格在 1920px 宽屏下实际可用宽度约 250px，但内容仅占顶部 40px（日期+统计标签+4个 6px 圆点），下方 48px 空白。任务超过 4 个时直接截断，无 `+N` 溢出提示，用户无法感知被隐藏的任务数量。

**V2. 任务圆点辨识度不足**

`w-1.5 h-1.5`（6px）的圆点在密集排列时几乎无法分辨颜色，尤其 pending（灰 `#909399`）与 skipped（灰 `#909399`）完全同色。圆点仅靠 `:title` 属性提供 hover 文案，移动端和触屏无法触发，信息获取成本高。

**V3. 状态色语义冲突**

`STATUS_META` 中 pending 和 skipped 共用灰色 `#909399`，两种状态在月视图圆点、周视图图标上无法区分。用户看到灰色圆点时无法判断是"待完成"还是"已跳过"，影响决策。

**V4. 周视图空间浪费**

`min-h-[280px]` 的列容器在任务少时留出大量空白，任务多时又因 `space-y-1` 和无滚动容器导致溢出。周视图缺少完成率统计、逾期汇总等概览信息，信息价值低于其占用的空间。

**V5. 加载状态缺失**

`loading` ref 已定义（第 50 行）但模板中从未消费。数据加载期间页面无任何反馈，用户点击"今天"或切换视图后会经历一段无响应的空白期，尤其在任务量大时体验割裂。

**V6. 空状态设计单薄**

日视图无任务时使用 `<NEmpty description="当日无任务" />`，视觉单调且无引导性。月视图/周视图在无任务日期仅显示数字或"无"字，缺少视觉引导。

### 2.2 交互逻辑维度

**I1. 月视图点击强制跳转打断浏览**

`selectDay`（第 157-164 行）在月视图下点击任意日期会立即 `viewMode.value = "day"` 切换到日视图。用户在浏览整月任务分布时，误点单元格就会被强制跳转，打断月视图的浏览节奏。周视图点击日期则无任何响应，交互行为不一致。

**I2. 无 hover 预览机制**

月视图任务点仅有原生 `title` tooltip，延迟约 1-2 秒且样式无法定制。用户想了解某日任务详情必须点击进入日视图，来回切换成本高。周视图任务行虽有 hover 背景色但无详情展开。

**I3. 批量操作仅限日视图**

全选/清空/批量完成/批量跳过四个按钮只在日视图 `header-extra` 出现（第 429-440 行）。用户在周视图看到某几天任务密集时，无法就近批量操作，必须切换到日视图逐天处理。

**I4. 无键盘导航支持**

方向键不能切换日期，Enter 不能进入日视图，Esc 不能返回上一视图。纯鼠标操作在频繁翻阅日期时效率低，尤其影响习惯键盘操作的用户。

**I5. 批量选择缺乏可视化进度**

`selectedTaskIds` 是 `Set<string>`，选中后仅在 checkbox 上反映，顶部无"已选 N 项"的实时计数。批量完成/跳过按钮也未根据选中数量动态禁用，空选时点击才提示"请先选择任务"。

**I6. 视图切换丢失上下文**

从月视图点击进入日视图后，返回月视图需要手动点击"月"按钮，且 `currentDate` 已变为点击的日期，原浏览位置丢失。缺少"返回月视图"的快捷回退路径。

### 2.3 功能完整性维度

**F1. 无任务筛选能力**

当目标数量增多后，日历视图无法按目标（goal）、状态（pending/done/skipped）、是否逾期过滤任务。用户被迫在密集的日视图中肉眼扫描，定位成本随任务量线性增长。

**F2. 无搜索定位**

缺少任务名搜索入口。周视图和月视图中找特定任务只能逐日翻阅，对于有几十个活跃目标的用户几乎是不可用的。

**F3. 无拖拽改期**

日历视图的核心价值之一是快速调整任务计划日期，当前仅支持查看与完成/跳过。改任务日期必须去目标树视图操作，日历沦为只读看板。

**F4. 无快速创建入口**

点击日期空白处不能新建任务，日历上无法为某天快速安排任务。创建任务仍需走目标树视图的目标拆分流程，与"日历安排"的心智模型割裂。

**F5. 月视图任务溢出无提示**

`getTasksOfDay(day).slice(0, 4)` 截取前 4 个任务，超出部分无 `+N` 标记。用户不知道某日实际有多少任务，可能低估负荷。

**F6. 无周/月统计概览**

缺少本周/本月完成率、逾期数、待办数的汇总展示。用户无法快速评估周期内的工作负荷与完成进度，需要心算统计标签。

### 2.4 性能与架构维度

**A1. 模板内重复计算**

`getDayStats(day)` 在月视图模板中被调用 4 次（第 339、342、345、350 行），每次渲染都重新遍历当日任务数组。`getTasksOfDay(day)` 同样在模板中被多次调用。任务量大时（如整月 200+ 任务）会触发不必要的重复计算。

**A2. 553 行单文件维护成本高**

月/周/日三套模板、批量操作逻辑、导航逻辑、数据加载全部内聚在一个文件中。修改任一视图都需要在长文件中定位，且难以复用子逻辑（如批量操作可跨视图共享）。

**A3. 无 composable 抽取**

导航（prev/next/goToday）、批量操作（batchComplete/batchSkip/selectAll）、数据加载（loadData/getDateRange）三组逻辑可独立为 composable，当前全部耦合在组件内，无法在其他视图中复用。

**A4. `watch` 触发粒度过粗**

`watch([viewMode, currentDate], loadData)` 在视图模式切换时会重新加载数据，但月→周切换可能查询的是同一日期范围（月初的周），产生冗余请求。

## 三、优化方案

按实施优先级分为四档。P0 为低风险快速修复，P1 为核心体验提升，P2 为能力扩展，P3 为架构演进。

### P0：快速修复（预计 0.5 天）

#### P0-1 加载状态可视化

**问题**：`loading` 已定义未使用，数据加载无反馈。

**措施**：在三种视图卡片内容区外层包 `NSpin :show="loading"`，月视图网格和日视图任务列表加载时显示旋转遮罩。导航按钮在 loading 时 `:disabled="loading"` 防止重复触发。

**预期效果**：消除"点击后无响应"的割裂感，操作反馈延迟感知从 1-2s 降至 200ms 内。

**涉及位置**：第 306、369、416 行的 `NCard` 内部。

#### P0-2 月视图任务溢出提示

**问题**：`slice(0, 4)` 截断后无 `+N` 提示。

**措施**：在任务圆点容器后追加溢出标记：

```html
<span v-if="getTasksOfDay(day).length > 4" class="text-[9px] text-gray-400">
  +{{ getTasksOfDay(day).length - 4 }}
</span>
```

**预期效果**：用户能感知当日实际任务量，避免低估负荷。

**涉及位置**：第 354-362 行任务点容器后。

#### P0-3 修复 skipped 状态色冲突

**问题**：pending 和 skipped 同为灰色 `#909399`。

**措施**：将 `STATUS_META.skipped.color` 调整为带斜线纹理或改为蓝灰 `#b0b3b8`，并配 `text-decoration: line-through` 视觉标识。在 `types/index.ts` 中修改：

```ts
skipped: { icon: "mdi:skip-next-circle-outline", label: "已跳过", color: "#b0b3b8" }
```

月视图圆点改用半透明 `opacity-50` 区分 skipped，pending 保持实色。

**预期效果**：消除灰色圆点的歧义，用户一眼区分"待完成"与"已跳过"。

**涉及位置**：`src/types/index.ts` 第 230 行、`CalendarView.vue` 第 359 行。

#### P0-4 批量操作按钮空选禁用

**问题**：空选时点击才提示"请先选择任务"。

**措施**：批量完成/跳过按钮绑定 `:disabled="selectedTaskIds.size === 0"`，并在按钮旁显示"已选 {{ selectedTaskIds.size }} 项"实时计数。

**预期效果**：减少无效点击，选中状态可视化。

**涉及位置**：第 432、436 行按钮。

### P1：核心体验提升（预计 2-3 天）

#### P1-1 月视图 hover 弹层预览

**问题**：任务点仅靠原生 `title`，信息获取成本高。

**措施**：用 `NPopover` 替换原生 tooltip，hover 日期单元格时弹出当日任务列表预览（前 8 条），每条显示状态图标+任务名+目标名。弹层内底部放"查看全部"按钮，点击才切换日视图。保留单元格 click 作为快捷入口，但改为 hover 800ms 后才弹出预览，避免误触。

```html
<NPopover trigger="hover" :delay="300" placement="bottom" :width="280">
  <template #trigger>
    <div class="calendar-cell" @click="selectDay(day)">...</div>
  </template>
  <div class="space-y-1">
    <div v-for="t in getTasksOfDay(day).slice(0, 8)" :key="t.id" class="flex items-center gap-2 text-xs">
      <Icon :icon="STATUS_META[t.status].icon" :color="STATUS_META[t.status].color" width="14" />
      <span class="flex-1 truncate">{{ t.name }}</span>
    </div>
    <NButton v-if="getTasksOfDay(day).length > 8" size="tiny" block @click="selectDay(day)">
      查看全部 ({{ getTasksOfDay(day).length }})
    </NButton>
  </div>
</NPopover>
```

**预期效果**：用户在月视图即可预览任务详情，减少不必要的视图切换，浏览效率提升 40%+。

**涉及位置**：第 319-364 行月视图单元格。

#### P1-2 周视图增加批量操作与统计

**问题**：周视图空间利用率低且无操作入口。

**措施**：
- 每列底部增加完成率进度条（`NProgress` size="small" :percentage="doneRate"）
- 每列任务行增加 checkbox，支持选中后顶部弹出批量操作工具栏
- 列头增加逾期数 `NTag type="error"` 标记
- 任务行支持点击展开详情（`NCollapse` 或 `NPopover`）

**预期效果**：周视图从"只读看板"升级为"可操作工作台"，减少向日视图的切换频率。

**涉及位置**：第 369-412 行周视图。

#### P1-3 顶部增加周期统计概览

**问题**：缺少周/月维度的完成率、逾期数汇总。

**措施**：在顶部工具栏下方增加一行统计卡片，根据当前视图模式展示对应周期的指标：

```
本周完成率 78% | 已完成 12 | 待完成 3 | 逾期 1
```

用 `NSpace` + 轻量统计组件实现，数值用 `text-xl font-bold text-brand-600`，标签用 `text-xs text-gray-500`。

**预期效果**：用户进入视图即知周期负荷与进度，无需心算。

**涉及位置**：第 303 行后插入统计卡片。

#### P1-4 缓存日期统计计算结果

**问题**：`getDayStats` 和 `getTasksOfDay` 在模板中重复调用。

**措施**：新增 `computed` 将日期到统计的映射预计算一次：

```ts
const dayStatsMap = computed(() => {
  const map: Record<string, ReturnType<typeof getDayStats>> = {};
  for (const day of monthGrid.value) {
    const key = format(day, "yyyy-MM-dd");
    map[key] = getDayStats(day);
  }
  return map;
});

function getDayStatsCached(day: Date) {
  return dayStatsMap.value[format(day, "yyyy-MM-dd")];
}
```

模板中将 `getDayStats(day)` 替换为 `getDayStatsCached(day)`。

**预期效果**：月视图渲染时计算量从 O(天数 × 调用次数) 降至 O(天数)，200 任务场景下渲染性能提升约 30%。

**涉及位置**：第 128-135 行 `getDayStats`、第 339-351 行模板调用。

### P2：能力扩展（预计 3-5 天）

#### P2-1 任务筛选栏

**问题**：无法按目标/状态/逾期过滤。

**措施**：在顶部工具栏增加折叠式筛选区，提供：
- 目标下拉多选（`NSelect multiple`，数据源 `goalStore.goals`）
- 状态多选（pending/partial/done/skipped）
- 逾期开关（`NSwitch`）

筛选条件变化时在前端过滤 `tasks.value`，不发起新请求。筛选状态用 `ref` 管理，重置按钮一键清空。

**预期效果**：多目标用户定位任务的时间从分钟级降至秒级。

**涉及位置**：第 283-303 行工具栏内新增筛选区。

#### P2-2 键盘导航

**问题**：纯鼠标操作效率低。

**措施**：
- 左右方向键：prev/next 切换日期
- 上下方向键（月视图）：在日期网格中移动选中格
- Enter：进入日视图查看详情
- Esc：从日视图返回上一视图模式
- `Cmd/Ctrl + A`：全选当日任务

用 `@keydown` 监听，仅在日历视图聚焦时生效（避免与全局快捷键冲突）。

**预期效果**：键盘用户操作效率提升，满足无障碍访问基本要求。

**涉及位置**：根容器增加 `tabindex="0"` 和 `@keydown`。

#### P2-3 月视图拖拽改期

**问题**：日历核心交互缺失。

**措施**：
- 月视图任务圆点改为可拖拽（`draggable="true"`，`@dragstart` 设置 `dataTransfer` 携带 `task.id`）
- 日期单元格作为放置目标（`@dragover.prevent`、`@drop` 调用 `taskApi.updateTaskDate`）
- 拖拽时源圆点 `opacity-40`，目标单元格 `ring-2 ring-brand-400`
- 改期成功后刷新数据并 toast 提示

需后端新增 `update_task_plan_date` 命令，或复用现有任务更新接口。

**预期效果**：补齐日历视图的核心编辑能力，从"只读看板"进化为"计划调整工作台"。

**涉及位置**：第 354-362 行任务点、第 319-329 行单元格。参考 `GoalTreeNodeItem.vue` 的拖拽实现（项目记忆中已记录 Tauri `dragDropEnabled: false` 约束）。

#### P2-4 快速创建任务

**问题**：日历上不能新建任务。

**措施**：日视图底部增加"快速添加"输入框，输入任务名 + 选择目标后调用 `taskApi.createTask`，`plan_date` 自动填充为当前选中日期。月视图双击空白日期单元格也可触发创建弹窗。

**预期效果**：补齐"计划安排"的创建闭环，用户无需离开日历视图即可安排任务。

**涉及位置**：第 549 行 `NEmpty` 附近增加创建入口。

### P3：架构演进（预计 2-3 天）

#### P3-1 组件拆分

**问题**：553 行单文件维护成本高。

**措施**：按视图拆分为独立组件：
- `CalendarMonthView.vue`：月视图模板+逻辑
- `CalendarWeekView.vue`：周视图模板+逻辑
- `CalendarDayView.vue`：日视图模板+逻辑
- `CalendarToolbar.vue`：顶部工具栏+统计+筛选

`CalendarView.vue` 退化为容器组件，负责数据加载、视图切换、状态管理。

**预期效果**：单文件行数从 553 降至约 150，各视图独立维护，修改影响范围可控。

#### P3-2 composable 抽取

**问题**：逻辑耦合无法复用。

**措施**：
- `useCalendarNavigation(viewMode, currentDate)`：封装 prev/next/goToday/headerLabel
- `useTaskBatch(tasks)`：封装 selectAll/clearSelection/batchComplete/batchSkip
- `useCalendarData(viewMode, currentDate)`：封装 loadData/getDateRange/tasksByDate

**预期效果**：批量操作逻辑可在月/周/日视图共享，消除重复代码。

#### P3-3 请求去重

**问题**：`watch` 切换视图模式时可能发起冗余请求。

**措施**：`loadData` 增加请求 token 机制，丢弃过期响应：

```ts
let loadToken = 0;
async function loadData() {
  const token = ++loadToken;
  loading.value = true;
  const { start, end } = getDateRange();
  const result = await taskApi.listTasksByDateRange(...);
  if (token !== loadToken) return; // 已被新请求取代
  tasks.value = result;
  loading.value = false;
}
```

**预期效果**：快速切换视图时避免旧响应覆盖新数据，消除偶发的数据闪烁。

## 四、实施路线图

| 阶段 | 内容 | 优先级 | 预计工时 | 风险 |
|------|------|--------|---------|------|
| 第一阶段 | P0-1 加载状态 + P0-2 溢出提示 + P0-3 状态色修复 + P0-4 按钮禁用 | P0 | 0.5 天 | 低 |
| 第二阶段 | P1-1 hover 预览 + P1-4 计算缓存 | P1 | 1 天 | 低 |
| 第三阶段 | P1-2 周视图增强 + P1-3 统计概览 | P1 | 1.5 天 | 中（周视图改动较大） |
| 第四阶段 | P2-1 筛选栏 + P2-2 键盘导航 | P2 | 2 天 | 中 |
| 第五阶段 | P2-3 拖拽改期 + P2-4 快速创建 | P2 | 3 天 | 高（需后端配合） |
| 第六阶段 | P3-1 组件拆分 + P3-2 composable + P3-3 请求去重 | P3 | 2.5 天 | 中（重构需回归测试） |

### 实施建议

**P0 阶段**改动小、收益明显，建议立即纳入下一次迭代。四项修改互不依赖，可并行开发。

**P1 阶段**是体验提升的核心，其中 hover 预览（P1-1）和计算缓存（P1-4）应优先实施，它们直接改善用户最常使用的月视图浏览体验。周视图增强（P1-2）改动较大，建议在 P1-1 验证后再推进，避免 popover 与周视图交互冲突。

**P2 阶段**涉及功能扩展，拖拽改期（P2-3）依赖后端新增接口，建议与后端同步排期。筛选栏（P2-1）和键盘导航（P2-2）可独立前端完成。

**P3 阶段**为架构优化，建议在 P1/P2 功能稳定后进行，避免重构与功能开发并行导致合并冲突。组件拆分后，后续迭代效率将显著提升。

### 验收标准

| 优化项 | 验收指标 |
|--------|---------|
| 加载状态 | loading 期间所有导航按钮禁用，Spin 遮罩覆盖内容区 |
| 溢出提示 | 月视图超过 4 任务时显示 `+N` 标记 |
| 状态色 | pending 与 skipped 在月视图圆点上视觉可区分 |
| hover 预览 | 鼠标悬停 300ms 弹出任务列表，含前 8 条任务 |
| 计算缓存 | 月视图 200 任务场景下首次渲染时间 < 50ms |
| 周视图批量 | 周视图可选中任务并批量完成/跳过 |
| 统计概览 | 顶部显示当前周期完成率、已完/待办/逾期数 |
| 筛选栏 | 可按目标+状态+逾期过滤，筛选结果实时更新 |
| 拖拽改期 | 月视图圆点可拖至其他日期，改期后数据刷新 |
| 组件拆分 | `CalendarView.vue` 行数 < 200，各子组件 < 300 行 |
